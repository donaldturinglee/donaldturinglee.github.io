正确使用评论。首先，对于给定的库、陈旭或函数，注释最好用来描述库、程序或函数的作用。它们通常放置在文件或库的顶部，或者紧接在函数之前
其次，在上述库、程序或函数中，注释可用于描述代码将如何实现其目标
第三，在语句级别，应该使用注释来描述代码执行某项操作的原因。一个糟糕的语句注释解释了代码正在做什么。如果写的代码非常复杂，需要注释来解释语句的作用，那么可能需要重写语句，而不是添加注释。

C++17**[[maybe_unused]]**属性，允许编译器可以接收未使用的变量

endl 和\n 的区别：
endl 会执行两项工作：1.将光标移动到控制台的下一行，并刷新缓冲区。
\n：将光标移动到控制台的一下行，但不刷新缓冲区（在输出文本到控制台时优先选择）

考虑将行的长度保持在 80 个字符或更少

Nullary 作用于零操作数

当函数参数存在但未在函数体中使用时，请勿为其指定名称。您可以选择在评论中输入名称。
out of scope: 标识符超出了该代码区域之外的范围
going out of scope: 通常适用于对象而不是标识符。对象在实例化对象的作用域末尾超出作用域

将局部变量定义为尽可能合理地接近其首次使用

使用函数的好处

  • 组织
  • 可重用性
  • 测试
  • 可拓展性
  • 抽象

编写函数的时机：

  • 在程序中出现多次的语句通常应该写成一个函数
  • 具有明确定义的输入和输出的代码
  • 一个函数通常应该仅执行一项任务
  • 当函数变得太长、太复杂或难以理解时，可以将它拆分为多个子函数

如果冲突的标识符被引入到同一个文件中，结果将是编译器错误。如果冲突标识符被引入到属于同一程序的不同文件中，则结果将是链接器错误
没有显示定义命名空间前缀时默认为全局命名空间
使用显式命名空间前缀来访问命名空间中定义的标识符。

预处理将源代码进行处理，生成翻译单元交由编译器进行编译
宏有两种基本类型：类对象宏和类函数宏

条件编译：允许在什么条件下编译或不编译某些内容
#ifdef、#ifndef 和#endif

C 风格：
#ifdef 允许预处理器检查标识符是否先前已被#定义
#ifndef 与#ifdef 相反，它检查标识符是否未被定义

C++风格：
#if defined()
#if !defined()

#if 0 是一种常见的方法来排除代码块的编译
#if 0
…
#endif

指令的定义仅从定义点到定义它们的文件末尾有效，一个文件中的指令定义不会影响项目中的其他代码文件

当#include 文件时，所包含文件的内容都将插入到包含点。这提供了一种从另一个文件读取声明的有用方法
头文件不应该添加到编译命令中，它们由#include 语句隐式包含并作为源文件的一部分进行编译）
避免将变量定义或函数定义放在头文件中。如果头文件包含在多个源文件中，这会违反单一原则（ODR）
不建议#include cpp 文件的几个原因：

  • 这样做可能会导致源文件之间命名冲突
  • 在大型项目中，很难避免单一原则（ODR）问题
  • 对某一 cpp 文件进行修改都会导致其他包含该 cpp 的文件进行重新编译，这可能会浪费很多时间。头文件的更改频率往往低于源文件

排错指南：如果编译器出错，检查头文件，如果链接器出错，检查源文件
<>和””的区别：

  • 使用尖括号时，告诉预处理器这不是我们写的头文件，需要搜索操作系统附带的头文件的目录，不会从项目目录中搜索头文件
  • 使用双引号时，告诉预处理器这是我们写的头文件，预处理器会首先在目录里搜索头文件，当在目录里搜索不到头文件时，它继续搜索操作系统附带的头文件的目录

如果没有.h 拓展名的头文件将名称声明为全局命名空间中，请避免使用这些名称，因为它们可能在其他编译器的全局命名空间中不可用。

对于包含其他目录中的头文件：

  • GCC/G++的做法： 使用-I/source/includes 指定头文件的目录进行编译。这种做法的好处是，如果更改了目录结构，则只需要修改单个编译器或 IDE 的设置，而不用更改源代码文件。注意：-I 后面没有空格。

每个文件都应该显示#include 需要编译的所有头文件，不要依赖于其他头文件传递地包含的头文件

加载头文件的方式：

  2. 项目中的其他头文件
  3. 第三方的头文件
  4. 标准库的头文件
    每个分组的标题应该按照字母排序

在头文件中编写有关某项功能或如何使用它的注释，描述某些内容如何工作的注释编写在源代码中

头文件保护：当一个文件包含同一个头文件多次时，会导致重复的声明和定义，从而引发编译错误。为了避免这种情况，通常在头文件的开头和结尾添加特定的预处理指令来实现。
具体流程：当头文件被 include 时，预处理会检查先前是否已经定义了标识符。如果这是第一次包含头文件，则标识符不会被定义。因此，它会#define 标识符并包含文件的内容。如果头文件再次包含在同一个文件中，则标识符从一次包含头文件内容时就已经被定义，并且头文件的内容将会被忽略（#ifndef)
命名约定：标识符设置为头文件的完整名称，全部大写，使用下划线表示空格或标点符号，最后加上下划线_H，例：

在同一目录下存在同名头文件的时候，可以在头文件保护中使用更复杂/唯一的名称来防止头文件保护名称冲突的可能性

头文件保护不会阻止头文件被包含到不同的源代码文件中，这有可能会导致链接时出现对标识符有多个定义。
解决方法：将函数定义放在其中一个 cpp 文件中，以便头文件仅包含声明

现代编译器使用#pragma 进行头文件保护，在传统的方式中，使用预处理器指令进行保护，而#pragma once 要求编译器保护头文件

设计程序的通用步骤：

  1. 定义目标：理想情况下应该能够用一两句话来表达这一点。将它表达为面向用户的结果通常很有用。例：
    • 允许用户组织姓名和关联电话号码的列表
    • 生成随机的场景，这些场景会生成看起来很有趣的东西
    • 生成高息股票的股票推荐列表
    • 模拟从塔上掉落的球到地所需的时间
      虽然这一步看起来很明显，但也非常重要
  2. 定义需求：虽然定义目标可以帮助确定想要的结果，但它仍然很模糊。下一步是考虑需求。需求是一个花哨的词，既指解决方案需要遵守的约束（例如预算、时间线、空间、内存等），也指程序为了满足用户需求而必须展示的功能。请注意，要求同样应该关注”什么“，而不是”如何“。例：
    • 应保存电话号码，以便以后调用
    • 场景应该始终包含入口到出口的方法
    • 股票推荐应利用历史定价数据
    • 用户应该能够输入塔的高度
    • 需要在 7 天内提供可测试的版本
    • 程序应该在用户提交请求后 10 秒内生成结果
    • 程序应该在不到 0.1%的用户会话中崩溃
      一个问题可能会产生许多要求，只有满足所有这些要求，解决方案才算”完成“
  3. 定义工具、目标和备份计划：此时通常会执行许多其他步骤
    • 定义程序将运行的目标架构和/或操作系统
    • 确定使用哪组工具
    • 确定是单独编写程序还是作为团队的一部分编写程序
    • 定义测试/反馈/发布策略
    • 确定如何备份代码
  4. 将困难问题分解为简单问题： 使用自顶向下的方法来解决问题，也就是说，我们不是解决单个复杂的任务，而是将该任务分解为多个子任务，每个子任务单独更容易解决。如果这些子任务仍然太难解决，可以进一步分解。通过不断地将复杂的任务分解为更简单的任务，最终可以达到每个单独的任务都可以管理的程度，尽管它所做的事情非常少。
  5. 弄清楚事件的顺序：当程序有了结构，就应该确定如何将所有的任务链接在一起。第一步是确定将执行的事件的顺序。

实现程序的通用步骤：

  1. 完成“大纲”：使用注释的方法完成程序的“大纲”，目前函数无法编译，因为定义还没有实现。在准备好实现函数定义之前注释掉函数调用是解决这个问题的一种方法
  2. 实现各个功能：在这个步骤中，对于每个函数，都将执行三件事情：
    1. 定义函数原型（输入和输出）
    2. 编写函数
    3. 测试功能
      如果函数足够精细，那么每个函数都应该相当简单明了。如果给定的功能看起来仍然过于复杂，那么可能需要将它分解为更容易实现的子功能。
  3. 最终测试：一旦程序“完成”，最后一步就是测试整个程序并确保它按预期工作。如果不起作用，请修复它

当出于调试目的打印信息时，请使用 std::cerr 而不是 std::cout。原因之一是 std::cout 可能会被缓冲，这意味着在要求 std::cout 输出信息和实际输出信息之间可能会有暂停。如果使用 std::cout 进行输出，但程序立即崩溃，则 std::cout 可能还没有实际输出。std::cerr 是无缓冲的，这意味着发送给它的任何内容都会立即输出
使用 std:cerr 还有助于明确输出的信息是针对错误情况而不是正常情况
添加临时调试语句时，不缩进它们会很有帮助。这使得以后更容易找到它们并删除。如果使用 clang-format 来格式化代码，可以使用

优化手动调试：使用预处理器指令使调试语句成为有条件的：

现在只需要注释/取消注释#define ENABLE_DEBUG 就可以启用或禁用调试。在多文件程序中，将它放入头文件中，以便可以在整个项目中使用。

使用日志进行调试：日志是已发生事件的顺序记录，通常带有时间戳。生成日志的过程称为日志记录。通常，日志会写入磁盘上的文件，以便稍后查看。大多数应用程序和操作系统都会写入可用于帮助诊断发生的问题的日志文件。
日志文件有一些优点。由于写入日志文件的信息与程序的输出是分开的，因此可以避免因混合正常输出和调试输出而造成的混乱。日志文件还可以轻松发送给其他人进行诊断。因此，如果使用软件的人遇到问题，可以要求他们发送日志文件，这可能有助于提供问题所在的线索。
使用 std::clog 写入日志信息。但默认情况下，std::clog 写入标准错误流（与 std::cerr 相同）。虽然可以将它重定向到文件，但在这方面，通常最好使用现有的第三方日志记录工具。例：plog、spdlog

step over 跳过函数的
step out 执行当前函数中所有剩余代码

在 C++中，通常使用“字节大小(byte)”的数据库
在新版本 C++中定义的许多类型都是用_t 后缀。这个后缀的意思是“类型”

void 类型：不完整类型（已声明但未定义的类型）；void 类型无法实例化
void 在上下文中的作用：

  • 不返回值的函数
  • 不带参数的函数（已弃用，在 C++中不推荐）

使用空参数列表而不是使用 void 来表明函数没有参数

对象使用的内存越多，它可以容纳的信息就越多。对于一个具有 n 位的对象（其中 n 是整数）可以保存 2 的 n 次方个唯一值。
计算机内存是有限的，每次定义一个对象时，只要该对象存在，就会使用一部分可用内存。

基础数据类型大小

为了可移植性，不应该假设变量大于指定的最小大小

为了确定特定机器上的数据类型的大小，需要使用 sizeof 运算符，它是一个一元运算符，它接收类型或变量，并返回其大小（以 byte 位单位）
sizeof 不能接收不完整类型，这将会导致编译错误；sizeof 不包括对象使用的动态分配的内存

默认情况下，C++的整数是有符号的，因此，有符号整数可以同时包括正数和负数（以及 0）
在二进制中，需要一个位（符号位）来存储数字的符号，剩余的位（幅度位）确定数字的幅度

在声明类型时，首选不使用 int 后缀或 signed 前缀的简写类型

对于有符号整数的范围：-(2^n-1) ~ 2^n-1 - 1

溢出：如果表达式在求值期间，结果未在数学定义上或不在其类型的可表示范围内，则行为未定义。一般来说，溢出会导致信息丢失

无符号整数时只能保存非负整数的整数。定义无符号整数使用 unsigned 关键字，它放在类型之前
1byte 无符号整数的取值范围为 0~255。有符号和无符号都可以存储 256 个不同的值，但有符号整数有一半的范围用于负数，而无符号整数可以存储两倍大的正数
无符号的取值范围为 0 ~ (2^n) - 1
当不需要负数时，无符号整数非常合适网络和内存很少的系统，因为无符号整数可以存储更多的正数而不占用额外的内存

无符号整数溢出：如果一个无符号值超出范围，它将被除以该类型的最大数加一，然后只保留余数。
另一种思考方式：任何大于该类型可表示最大数的数字都会简单地“环绕”。例：256 超出了范围，因此它会环绕到值 0，257 会环绕到 1
当然也可以绕向另一个方向。例：-1 会环绕到 65535， -2 会环绕到 65534

避免使用无符号整数：

  • 溢出会比有符号整数容易的多，因为绝大多数值都非常靠近 0
  • 当使用自减时的值变得小于 0 时，会发生另一种常见的不需要的环绕。
  • 当有符号和无符号混合运算时时，会转换成无符号整数进行运算，结果将是无符号整数

建议：在保存数量（甚至应该是非负的数量）和数学运算时，优先选择有符号数而不是无符号数。避免混合有符号数和无符号数

无符号整数的使用场景：

  • 处理位操作

固定宽度整数（保证在任何体系结构上具有相同的大小）：

在 C++中导入 stdint 头文件

缺点：

  • 不保证在所有体系结构上都定义固定宽度整数。他们仅存在于具有和他宽度匹配的基本类型并遵循某种二进制表示形式的系统上
  • 如果使用固定宽度的整数，在某些体系结构上它可能会比更宽的类型慢

为了解决上面的缺点，C++还定义了两组备用的类型整数，这些类型被保证是已定义的
std::int_fast#_t 和 std::uint_fast#_t 提供最快的有符号/无符号整数类型，宽度至少为#位(#：8，16，32，64)。所谓最快，指的是 CPU 处理速度最快的整型
std::int_least#_t 和 std::uint_least#_t 提供最小的有符号/无符号整数类型，宽度至少为#位(#：8，16，32，64)

在存储整数值时最好避免 std::int8_t 和 std::uint8_t（以及相关的 fast 和 least 类型）并使用 std::int16_t 或 std::uint16_t 代替。因为大多数编译器将 std::int8_t 和 std::uint8_t（以及相关的 fast 和 least 类型）分别定义和处理为 signed char 和 unsigned char 类型。

8 位固定宽度整数类型通常被视为字符而不是整数值（因系统而异）。大多数情况下首选 16 位固定整数类型

整数类型使用建议：

  • 正确比更快的好
  • 编译时失败比运行时失败好

在 C++中使用 e 来表示方程乘以 10 次方部分

在科学计数法中，如果数字是需要精确测量的，那么不应该移除小数点后的尾随零

在定义浮点数时，始终至少包含一位小数（即使小数为 0），这有助于编译器理解该数字是浮点数而不是整数
默认情况下，浮点数为 double 类型，f 后缀用于表示 float 类型

建议： 始终确保文字的类型与他们被分配或初始化的变量类型相匹配。否则将导致不必要的转换，并可能导致精度损失
输出浮点数时，cout 默认精度为 6
浮点类型的精度为 6-9 位，有效位至少有 7 位。
Double 类型精度位 15-18 位，有效位至少有 16 位
Long Double 的最小精度为 15、18 或 33 个有效数字，具体取决于它占用的字节数

在 C++中，可以使用**std::setprecision()**覆盖默认显示的精度，它需要引入 iomanip 头文件

浮点数有两种特殊类别。第一个是 Inf 代表无穷大。Inf 可以是正值也可以是负值。第二个是 NaN，代表不是“数字”。NaN 有几种不同的类型。仅当编译器对浮点数使用特定格式（IEEE 754）时。NaN 和 Inf 才可用。如果使用其他格式，会产生未定义的行为

在 C++中，如果期望打印的 Boolean 值是 true 或 false，可以使用 std::boolalpha（只接受小写的 false 和 true）
在 Boolean 判断中，0 会转换为 false，非 0 都会转换为 true
布尔值通常用作检查某事是否正确的函数的返回值。此类函数的命名通常以单词is或has开头

if 语句和提前 return：不是函数中最后一条语句的返回语句称为提前返回。这样的语句将导致函数在执行 return 语句时返回给调用者

没有返回值的提前返回是没有用的

在早期的编程中，提前返回是不被允许的。然而在现代编程中，它们更容易被接受，特别是当它们可以用于使函数更简单，或用于由于某些错误条件而提前终止函数时

在字符中，32~127 称为可打印字符，多用于显示基本英文文本的字母、数字字符和标点符号
当想要将数字表示为文本而不是要应用于数学运算的数字时，应该使用字符数字

将独立字符放在单引号中，这有助于编译器更有效地优化
避免使用多字符文字
在 C++中避免使用 wchat，推荐使用 char16_t 和 char32_t，它们提供对 16 位和 32 位 Unicode 字符的显示支持。

当编译器在没有明确询问的情况下为我们进行类型转换时，称之为隐式类型转换。尽管称为转换，但类型转换实际上不会更改值或转换值的类型

通过 static_cast 进行显示类型转换： static_cast(expression)。
每当看见使用尖括号的 C++语法时（不包括预处理器），尖括号之间的内容很可能是一种类型。这通常是 C++处理需要参数化类型的代码的方式
注意：如果要转换的值不合适新类型的范围，static_cast 运算符将产生未定义的行为

按值传递时不要使用常量函数参数
按值返回时，const 值会阻碍某些类型的编译器优化（涉及移动语义），从而导致性能降低
优先使用常量变量而不是宏定义的原因：

  • 宏不遵循正常的 C++作用域规则：宏会全局替换，但在某些地方不需要进行宏替换，这可能会导致奇怪的编译错误
  • 宏调试代码更困难：因为运行之前宏都会被替换为具体的值，但编译器和调试器永远不会看到宏，并且许多调试器无法检查宏的值和在使用宏时通常功能有限
  • 宏替换的行为与 C++中其他所有内容不同。结果很容易犯无心的错误

类型限定符：const 和 volatile
volatile 限定符：用于告诉编译器一个对象的值可以随时修改。这个很少使用的限定符会禁用某些类型的优化

as-if 规则：假设规则表示，编译器可以随意修改程序，以生成更优化的代码，只要这些修改不会影响程序的“可观察行为”
有一个例外：对复制构造函数的不必要的调用可以被忽略，即使这些复制构造函数具有可观察到的行为。

常量表达式：编译器在编译时可以计算的表达式。要称为常量表达式，表达式中所有值必须在编译时已知（并且调用的所有运算符和函数必须支持编译时求值）
优点：表达式只需要计算一次（而不是每次运行程序时）。生成的可执行文件速度更快并且使用更少的内存
缺点：编译时间变长

运行时常量和编译时常量
编译时常量：它的值为常量表达式的常量。字面量是编译时常量的一种类型
const 变量可能是也可能不是编译时常量（取决于它们的初始化方式）
编译时 const 变量通常作命名常量
优化编译时常量：例：每次使用编译时常量时，编译器都可以简单地将标识符替换为字面量，这样避免了必须获取来自内存中某处的值
使变量成为编译时常量有助于编译器确定可以优化的内容

运行时常量：直到运行时才能确定其初始化值的常量
运行时常量只能在非常量表达式中使用。它们的主要用途是确保对象的值不被修改

constexpr：指明变量只能是编译时常量。如果 constexpr 变量的初始化值不是常量表达式，编译器会抛出错误

建议：

  • 任何初始化后不应该修改且其初始值设定项在编译时已知的变量都应声明为 constexpr
  • 任何在初始化后不应该修改且其初始值设定项在编译时未知的变量应声明为 const

函数参数不能声明为 constexpr
编译器一直能够优化常量子表达式，即使完整表达式是运行时表达式也是如此。这个优化过程被称为“不断折叠”。将变量设置为 constexpr 可以确保这些变量在常量子表达式中使用时符合常量折叠的条件

字面量：指的是在源代码中直接表示固定值的标记。这些值可以是整数、浮点数、字符串、字符等
字面量后缀：

优先使用字面量后缀大写 L 而不是小写 l
std::string 和 std::string_view 创建的是临时对象，而传统 C 字符串创建的是 const 对象，并保证在整个程序中存在

幻数：一个字面量（通常是数字），其含义不明确或者可能需要稍后修改
符号常量可以轻松解决上下文缺乏和更新问题
常量的名称提供了上下文，只需要更新一个位置的值即可在整个程序中更新该值
注意：幻数不总是数字，它们可以是文本或其他类型
避免在代码中使用幻数（使用 constexpr 变量代替）

在 C++14 中，可以使用 0b 前缀来使用二进制文字
C++14 中可以添加’作为数字分隔符的功能。（分隔符不能出现在值的第一个数字之前），分隔符纯粹是视觉上的，不会以任何方式影响字面量值
std::dec 输出十进制（默认）
std::oct 输出八进制
std::hex 输出十六进制
std::bitset 定义 std::bitset 变量并告诉它需要存储多少位。位数必须是编译时常量。它可以使用整数值（任何格式，包括十进制、八进制、十六进制或二进制）进行初始化

条件运算符（三元运算符）： condition ? expression1 : expression2;

由于条件运算符的操作是表达式而不是语句，因此可以在需要表达式的地方使用条件运算符
建议：在符合表达式中使用条件运算符时，将整个条件运算符括起来；为了方便阅读，如果条件包含任何运算符（函数调用运算符除外），用括号括起来
在使用条件运算符时，保证类型匹配，否则可能会出现意想不到的结果
最好避免在复杂表达式中使用条件运算符

内联扩展：将一个函数调用替换为被调用函数定义中的代码过程
内联代码的性能：除了消除函数调用的成本之外，内联扩展还可以让编译器更有效地优化结果代码
缺点：如果被拓展的函数体比被替换的函数调用需要更多的指令，那么每次内联扩展都会导致可执行文件变大。较大的可执行文件往往速度较慢（由于不太合适内存缓存）
建议：内敛扩展最合适简单、短的函数（不超过几条语句），特别是单个函数调用可以执行多次的情况（例如循环内的函数调用）

现代优化编译器决定何时应用内联扩展函数（不要使用 inline 关键字来请求函数的内联扩展）

在现代 C++中，inline 演变成了允许多个定义的意思。因此，内联函数是一种允许在多个文件中定义的函数

内联函数有两个主要要求：

  • 编译器需要能够在使用该函数的每个翻译单元中查看内联函数的完整定义（提前声明本身是不够的）。如果还提供了提前声明，则定义可以在使用点之后发生
  • 内联函数的每个定义都必须相同，否则将导致未定义的行为

连接器会将标识符的所有内联函数定义合并为单个定义

constexpr 函数可以在编译时计算其返回值的函数。在函数返回类型前添加 constexpr 关键字即可

如果在需要常量表达式的地方使用返回值，则可以在编译时计算的 constexpr 函数仅在编译时计算。否则无法保证编译时评估。因此，constexpr 函数最好被认为是“可以在常量表达式中使用”，而不是“将在编译时求值”

在 C++20 中，consteval 关键字表示函数必须在编译时求值，否则会导致编译错误。此类函数称为立即函数
建议：如果有一个函数因为某种原因（例如性能）必须在编译时运行，使用 consteval

按值传递：编译器实际上是将参数的值副本传入函数中，而不是参数本身。当函数内创建的所有参数副本和变量都将被销毁。

传递引用：传递变量的地址而不是变量本身，这将确保函数操作实际变量

函数重载：使函数根据其参数的性质执行不同的操作（没有参数且仅返回类型不同的函数不能重载，因为编译器无法区分它们的调用）
函数重载可以节省内存，编译速度更快（特别是在参数列表很长的情况下），代码更简单干净

在 C++中，栈代表 RAM 的静态部分。它以栈形式存储不同的函数（最近调用的函数在顶部）以及其中使用的所有变量
每当一个函数被释放，它在内存中的空间就会被释放，并且这些变量将永远丢失。
仅当该函数位于栈顶部（正在执行的函数）时，才能访问该函数的成员

内存动态部分称为堆。它是一个不受 CPU 管理的巨大内存空间
在堆上，可以指定我们想要的空间量，并且堆的随即部分将会保留。与栈相比，堆允许使用尽可能多的空间，但查找操作速度较慢。
堆中的变量可以具有“全局”作用域名，而不仅仅是函数的局部变量
堆是在运行时创建的（不需要在编译时制定所需要的内存空间），栈是在编译时创建的

将指针传递给函数，则该函数可以直接操作指针指向的值。从这个意义上说，指针可以充当可由所有函数访问的全局变量
由于数组是指向内存块的指针，因此它们也通过函数的引用进行传递。不需要使用**&**运算符

指针支持：**++、–、+=、-=**

指针不能相加，两个指针可以相减，结果是一个整数（也可能是一个长整数）

类对象使用**.**运算符访问类的公共成员
protected 主要用途是实现继承，它介于 private 和 public 之间
始终小心类里的数组，应当进行边界检查，确保不会产生数组越界错误。一种好的做法是将数组的大小存储在变量中

**::**作用域解析运算符：简单地声明类中的成员函数并使用以下语法在代码中的其他位置定义它们

returnType className::function()

**::**运算符告诉编译器该特定函数属于它前面的类
在 OOP 中，set 函数设置值，get 函数检索特定数据成员的值

构造函数：特殊的成员函数，它描述类在程序中创建类的实例时执行的步骤
在默认构造函数中，定义类的数据成员的默认值。因此，构造函数创建一个对象，其中的数据成员被
初始化为其默认值

this指针指向调用对象本身。可以使用this指针来访问调用对象的数据成员。

析构函数：当类的对象不再使用时调用它。该对象被销毁，它占用的内存现在可以释放以供将来使用
在 C++中，为了有效地释放内存，必须为类指定自己的析构函数。在析构函数中，可以指定删除类对象时需要执行的附加操作
析构函数的创建方式与构造函数类似，只是析构函数前带有**~**关键字

显示垃圾收集：通过智能指针可以轻松实现小程度的垃圾收集。智能指针，特别是shared_ptr，保留它所指向的对象的引用计数。当计数器降到 0 时，该对象被删除。

对于对象指针，当使用 delete 命令时，会调用析构函数

如果不为未使用的对象释放内存，最终将导致内存泄漏，并且应用程序没有空间长期运行

友元函数： 一个独立的函数，可以访问其友元类的变量和方法。（访问不属于类的函数中的类变量）
创建友元函数，必须在类中与friend关键字一起声明它。
方法可以独立在类之外，但它可以访问所有私有变量

数据隐藏：隐藏类内部结构的概念。提供了一个接口，外部访问可以通过该接口与类进行交互，而无需了解其内部细节。

数据隐藏可以分为两个主要部分：抽象和封装

数据封装：私有变量不能直接从外部访问，但提供类允许访问这些变量的读写函数
封装的优点：类更容易更改和维护; 可以指定要隐藏或可访问的数据成员; 决定哪些变量具有读/写权限(增加灵活性)

抽象：侧重于仅揭示应用程序的相关部分，同时隐藏内部实现。
只有输入和输出才最重要，不关心程序内部执行了哪些计算，这就是使用类进行抽象的过程

类继承的三种模式： public、private和protected。
继承的基本语法：

class derivedClassName : modeOfInheritance baseClassName

基类构造函数：当创建不带参数的子类实例时，它将首先调用父类的默认构造函数，然后调用子类。同样的，当调用子类的有参构造函数时，它会先调用父类的有参构造函数，然后调用子类的有参构造函数

基类析构函数：当创建子类的实例时，它将首先调用子类的析构函数，然后调用父类的析构函数

函数重写：子类继承父类时，可能会选择更改某些继承的功能。这称为函数重写，因为子类重写了父类的函数。

重写子类中的函数会隐藏父类中同一函数的所有重写

private 继承方式：父类的 private 数据成员和成员函数在子类中是不可访问的，子类可以访问父类的 protected 和 public 成员，并且行为与子类的 private 成员相同

从上述子类继承的任何类无权访问父类

protected 继承方式：父类的 private 成员在子类中不可访问。子类可以访问父类的 protected 和 public 成员，并且行为与子类的 protected 成员相同

public 继承方式：父类的 private 成员在子类中不可访问。子类可以访问父类的 protected 成员，并且行为与子类的 protected 成员相同。子类可以访问父类的 public 成员，行为与子类的 public 成员相同

  • 子类中的函数必须具有彼此不同的实现
  • 父类中的方法必须需要在子类中重写

virtual 函数：Virtual 是指存在于表象而非现实中。
virtual 函数是在父类中声明并由子类重写的成员函数。当使用指针或对父类的引用来引用子类对象时，可以调用该对象的virtual函数并执行该函数的子类版本

它们主要用于实现运行时多态性。函数在父类中使用virtual 关键字声明。函数解析调用是在运行时完成的。

抽象类：只能让子类的对象访问它们的函数，而永远不会想要实例化父类的对象。这样的类的存在只是为了充当用于实例化对象的子类的父类

pure virtual 是在声明中添加表达式**=0**的函数

=0 符号

composition 正在访问类中的其他对象，所有者类拥有该对象并对其生命周期负责。

在组合中，父对象包含子对象，这将父子联系起来。
在聚合中，父对象仅包含对子对象的引用，这消除了子对象的依赖性

在聚合中，拥有的对象的生命周期不依赖于所有者的生命周期

需要使用指针来实现聚合

传统测试和单元测试
在传统测试中：

  • 单个组件很少经过测试
  • 错误未被发现
  • 错误的隔离很难追踪

传统测试策略：

  • 打印语句
  • 使用 debug
  • debug 表达式
  • 测试脚本

缺点：
传统的测试策略不足以测试单个单元。

在单元测试中：

  • 每个组件都经过单独测试
  • 所有组件至少测试一次
  • 错误会被更早地发现
  • 范围较小；因此，更容易修复错误

优点：

  • 调试效率更快
  • 开发效率更高
  • 设计上更好
  • 有很多优秀的工具
  • 降低长期开发成本

位操作

优点：动作快速而简单，直接由处理器支持，用于操作值进行比较和计算，位运算符比算术运算符简单且高效
位操作符都是从右到左进行计算
右移公式：将数字除以$2^k$,相当于num >> 1 $\frac{num}{2}$
左移公式：将数字乘以$2^k$,相当于num << 1 $1 * 2^k$

例：二进制转换，二进制位数计算

& ：如果两个输入位均为 1,则输出 1; 在其他所有情况下为 0。

语法： a & b

AND 将第一个操作数的巍每一位与第二个操作数的每一位进行相应位比较。

任意数字 x 与 0 进行&操作会得到 0（相当于将它们相乘）

AND 运算符执行的操作：

  • 二元运算符，需要两个数字
  • 当对这两个数字进行 AND 操作时，它会考虑这两个数字的二进制表示形式
  • AND 以二进制表示形式进行逐位比较，无论得到什么二进制表示形式，都会返回相应的整数作为输出

例：计算二进制有效位(布莱恩算法)

现代操作系统: 一个复杂的实体. 它具有附加功能, 如统计数据收集, 多媒体处理, 系统安全保障, 整体稳定性, 可靠的错误处理等

操作系统提供 API 的原因:

  • 虽然操作系统有义务执行所有这些任务, 但编程人员仍有必要注意系统的细节和要求. 例如, 通过虚拟机从更高的抽象层次进行工作并不意味着放弃了解代码如何影响系统行为的需要. 更接近操作系统层的编程者也需要有效地管理系统资源

类 Unix 操作系统被视为多用户, 因为多个用户可以在系统上同时工作, 其中每个 shell 或终端实例都被视为单独的用户会话
通用操作系统(GPOS)
实时操作系统(RTOS): 分为硬 RTOS 和软 RTOS

用户是 GPOS 系统功能的主要驱动力. 操作系统的主要工作是保持与用户的主动对话以及操作的高可用性
每个任务和对操作系统的每个请求都必须在严格的时间间隔内快速处理. RTOS 仅在异常情况\错误和不可预测的行为期间期望用户输入

Linux 内核的三个主要层
用户空间: 正在运行的进程及其线程
内核空间: 正在运行的内核本身, 通常是它自己的进程
计算机: 可以是任何类型的计算设备, 如: PC, 平板电脑, 智能手机, 超级计算机, 物联网设备等.

使用系统调用来实现或执行操作系统提供的功能称为系统编程
常见的系统函数

  • syscall()
  • fork()
  • exec()
  • exit()
  • wait()
  • kill()

三个重要术语: 文件, 进程和线程

文件

  • 代表系统上的多种资源
  • 实现通信机制和存储管理

Linux 上可以识别的文件类型

  • 普通或常规文件: 系统上几乎所有存储数据的文件都被视为常规文件: 文本, 媒体, 代码等
  • 目录: 用于构建系统的层次结构. 不存储数据, 而是存储其他文件的位置
  • 特殊(设备)文件: 可以在 /dev目录下找到, 代表所有的硬件设备
  • 链接: 使用它们来运行访问不同位置的另一个文件. 实际上, 它们是真实文件的替换, 通过它们可以直接访问这些文件.(与 Windows 的快捷方式不同, 它们是特定的文件类型, 需要应用程序来支持它们 - 首先处理快捷方式元数据, 然后指向资源, 因此文件不会一次性被访问)
  • sockets: 进程交换数据(包括与其他系统)的通信端点
  • 命名管道(named pipes): 使用命名管道在系统上当前运行的两个进程之间交换双向数据

进程: 程序的一个实例(一个执行实例). 它有自己的地址空间并与其他进程保持隔离. 进程表通过每个进程的特定进程标识符进行哈希处理 - pid

快速查找进程

一个新的进程是通过复制当前进程的属性而创建的, 并将属于一个进程组. 一个或多个组形成一个会话. 每个会话与一个终端相关联. 组和会话都有进程领导者. 属性的 clone 主要用于资源共享. 如果两个进程共享相同的虚拟内存空间, 它们被视为单个进程中的两个线程, 但它们不像进程那样沉重

四个实体:

  • 可执行文件 : 要执行的指令的单元载体
  • 进程: 执行指令的工作单元
  • 指令: 指令作为处理和管理系统资源的工具
  • 线程: 最小的指令序列,由操作系统独立管理,是进程的一部分

fork() : 系统函数, 用于创建进程调用者的进程副本

前台进程: 有些进程需要用户交互才能启动或进行交互
后台进程: 独立于任何用户的活动而运行
终端输入的程序执行调用或用户命令默认为前台进程, 要在后台运行一个进程, 只需要在用于启动进程命令行末尾加上 & 符号

终止进程

kill pid

人为终止进程和程序自行终止进程是两件不同的事情. 人为终止进程可能会导致不可预测的系统行为或无法访问某些资源, 例如未关闭的文件或 socket

守护进程: 在无人值守的情况下在后台持续运行. 守护进程通常通过系统的启动脚本启动并运行到直到系统关闭. 他们通常提供系统服务, 并且多个用户依赖它们. 因此, 启动时的守护进程通常由 ID 为 0 的用户(通常是 root )启动, 并且可能以 root 权限启动

僵尸进程: 已终止但仍可以通过 pid 识别的进程. 它没有地址空间, 只要其父进程运行, 僵尸进程就会继续存在. 这意味着, 在退出主进程, 关闭系统或重启系统之前, 僵尸进程在 ps 列出时仍将显示为

init : 初始进程, 在 Linux 系统上由内核执行, pid 始终为 1

init 被称为系统上所有进程的父进程, 因为它用于初始化, 管理和跟踪其他服务和守护进程.

systemd 是一个 init 守护进程,它代表了在 Linux 系统上运行服务的现代方法.
每个服务都存储在 /lib/systemd/system 或 /etc/systemd/system 目录下的 .service 文件中

/lib 中的服务是系统启动服务的定义
/etc 中的服务是系统运行时启动的服务的定义

在 systemd 目录中, 有许多类型的文件. 在守护进程的上下文中, 它们被称为 units
.timer 用于计时器管理
.service 用于指定给定服务的启动方式及其依赖项
.path 描述给定服务的基于路径的激活服务等

便携式操作系统接口(POSIX): 主要任务是维护不同操作系统之间的兼容性(使用 POSIX, 编程者不需要区分系统调用和库函数)

POSIX API: 在系统编程的几个重要领域中, 为系统调用接口提供了附加功能:文件操作, 内存管理, 进程和线程控制, 网络和通信以及正则表达式

POSIX 软件需求中的一个著名概念: 一次编写, 随处使用

该标准定义并针对其应用的四个主要领域,称为卷

  • 基本定义: 语法, 概念, 术语和服务操作
  • 系统接口: 接口描述和定义的可用性
  • 实用程序: Shell, 命令和实用程序描述
  • 原理: 版本信息和历史数据

内存段: 内存布局或内存部分. 这些只是内存区域, 不应与分段存储体系结构混淆
主要的内存段包括文本(或代码), 数据, BSS, 栈和堆, 其中 BSS 代表由符号开始的块或块起始符号

文本：将在计算机上执行的代码. 它是在编译时创建的. 在运行时, 它是进程的只读部分. 当前的机器指令可以在这里找到, 根据编译器的不同, 也可以在这里找到 const 变量
数据: 该段也在编译时创建，由初始化的全局数据、静态数据或全局数据和静态数据组成。当您不想依赖运行时分配时，它用于初步分配的存储。
BSS: 与数据段相反，BSS 不在目标文件中分配空间 - 它仅在程序进入运行时标记所需的存储空间。它由未初始化的全局数据、静态数据或全局数据和静态数据组成。该段是在编译时创建的。理论上，根据语言标准，其数据被视为初始化为 0 ，但实际上在进程启动期间由操作系统的程序加载器设置为 0
栈: 程序栈是代表正在运行的程序例程的内存段 - 它保存其局部变量并跟踪被调用函数返回时从何处继续. 它在运行时构建并遵循 LIFO 策略. 我们希望保持它小而快
堆: 另一个运行时创建的段, 用于动态内存分配. 对于许多嵌入式系统来说, 它被认为是禁止的

分析一个二进制对象使用 size 命令

dec : 总体大小(bytes)
text : 指令(bytes)
data : 数据(bytes)

使用 readelf -s test 在需要时进一步分析

未初始化的静态变量必定会导致 BSS 增长

Linux 任务状态

  • Executing (R - Running and Runnable) : 为进程的指令提供处理器(核心或线程) - 任务正在运行. 调度算法可能会强制它执行. 然后, 任务变得可运行, 并被添加到可运行队列中, 等待轮到它们. 两种状态时不同的, 但都表示为正在执行的进程
  • Sleeping (D - Uninterruptible and S - Interruptible) : 因等待外部资源而导致的不间断睡眠, 睡眠不能通过信号中断, 直到资源可用并且进程可用再次执行为止. 可中断睡眠不仅依赖于资源可用性, 而且允许进程由信号控制
  • Stopped (T) : 将进程置于停止状态的信号, 但根据信号请求, 它可能会被忽略, 并且进程将继续. 或者, 可用停止该过程, 直到收到再次继续的信号为止
  • Zombie(Z) : 进程以终止, 但它在操作系统的任务向量中仍然可见

显示每个进程状态的字母表示, 其中 STAT 列提供当前状态

常见的调度算法

  • 先来先服务(FCFS)
  • 最短作业优先(SJF)
  • 最高优先级优先(HPF)
  • 循环(Round-robin)
  • 完全公平调度(CFD) : 目前 Linux 的调度机制, 它根据系统状态应用上述算法的组合

chrt -m : 查看调度算法

  • 优先级(priority): 它的值是实际的任务优先级, 用于调度. 0-99 之间的值专用于实时进程, 而 100-139 之间的值用于用户进程
  • nice: 它的值在用户空间级别有意义, 并在运行时调整进程的优先级. root 用户可以将其设置为-20-19, 简单用户可用将其设置为 0-19, 较高的 nice 值意味着较低的优先级. 默认为 0

对于用户进程, 优先级=nice+20
对于实时进程, 优先级=-1-real_time_priority
进程优先级越高,调度优先级越低
无法更改进程的基本优先级, 但可用使用不同的 nice 值启动它
